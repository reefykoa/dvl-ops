Perform a security-focused audit of the changes in a GitHub PR and report findings by severity.

## Step 1 — Collect inputs

Use `AskUserQuestion` to ask: "What is the GitHub PR URL?"

Use `AskUserQuestion` to ask: "Any specific security areas to focus on? (e.g. authentication, API keys, SQL injection, XSS) — or leave blank for a full review."

## Step 2 — Fetch the PR

Parse `owner`, `repo`, and PR number from the URL.

Run:
```
gh pr view <number> --repo <owner>/<repo> --json title,body,files
gh pr diff <number> --repo <owner>/<repo>
```

For each changed file, read its full current contents to understand the context around the diff.

## Step 3 — Evaluate for security issues

Review the diff and changed files against the following categories. If the user specified focus areas in Step 1, prioritize those — but still scan all categories.

### A. Injection (OWASP A03)
- SQL injection: user input concatenated into SQL queries without parameterization
- Command injection: user input passed to shell commands, `exec`, `eval`, or subprocess calls
- LDAP / XPath / NoSQL injection: similar patterns in other query languages

### B. Broken Authentication (OWASP A07)
- Hardcoded credentials, API keys, tokens, or passwords in source code
- Weak or missing authentication checks on endpoints or functions
- Insecure token storage (e.g. storing auth tokens in localStorage, plain text, or logs)
- Missing session invalidation on logout

### C. Sensitive Data Exposure (OWASP A02)
- PII, payment data, or secrets logged to console or written to files unencrypted
- Sensitive data returned in API responses that should be filtered
- Weak cryptography (MD5, SHA1 for passwords, hardcoded IVs)

### D. Broken Access Control (OWASP A01)
- Missing authorization checks — any user can access another user's resources
- Privilege escalation paths — low-privilege user can reach admin functions
- Direct object references without ownership validation

### E. Security Misconfiguration (OWASP A05)
- Debug mode or verbose error messages exposed in production paths
- CORS set to `*` without justification
- Missing security headers (CSP, HSTS, X-Frame-Options) where they were previously present

### F. Cross-Site Scripting — XSS (OWASP A03)
- User input rendered into HTML without escaping
- `dangerouslySetInnerHTML` or equivalent used with unsanitized data
- Inline event handlers built from user-controlled strings

### G. Insecure Dependencies
- New packages added to the dependency manifest
- Flag these for the user to check with `npm audit`, `bundle audit`, or equivalent — do not attempt to look up CVEs directly

### H. Insufficient Logging / Monitoring (OWASP A09)
- Security-relevant events (failed logins, permission denials, data deletions) with no logging
- Existing audit logs removed or bypassed

### I. Input Validation
- Missing validation at system boundaries: API endpoints, form inputs, file uploads, URL parameters
- Trust boundaries crossed without sanitization (e.g. data from external API used directly in DB query)

### J. GitHub Actions / CI Security (if workflow files are changed)
- Secrets printed to logs
- Untrusted inputs used in `run:` steps without sanitization (script injection)
- Overly broad permissions (`permissions: write-all`)
- Third-party actions pinned by mutable tag instead of commit SHA

## Step 4 — Classify and report findings

Classify each finding:
- **High** — directly exploitable, data loss risk, authentication bypass, secret exposure
- **Medium** — security weakness that requires additional conditions to exploit, or defense-in-depth concern
- **Low** — best practice violation, minor hardening opportunity, informational note

Present findings in this format:

```
## Security Review: PR #<number>

### High Severity
- **[Category]** `file.ts:42` — <description of the issue and why it is a risk>
  *Recommended fix:* <specific, actionable guidance>

### Medium Severity
- ...

### Low Severity
- ...

### Dependency Flags (check manually)
- `package.json`: new package `<name>@<version>` added — run `npm audit` to check for known CVEs

---
*No issues found in a category = that category is not listed.*
*Analysis generated by Claude — verify before acting.*
```

If no issues are found in any category: print "No security concerns identified in this diff."

## Step 5 — Post to PR (if requested)

Use `AskUserQuestion` to ask: "Should I post this security review as a comment on the PR?" with options: Yes / No.

If Yes:
1. Write the report to `/tmp/security-review-<PR_NUMBER>.md`
2. Post:
   ```
   gh pr comment <number> --repo <owner>/<repo> --body-file /tmp/security-review-<PR_NUMBER>.md
   ```
3. Output the URL of the posted comment.

## Rules

- Only report findings with clear evidence in the diff — do not speculate about code not changed in this PR.
- Do not apply fixes automatically. Security changes require human review before committing.
- Do not raise false alarms on patterns that are safe in context — explain your reasoning when a finding depends on context.
- Hardcoded secrets are always High severity regardless of context.
- If a changed file is a test file and the "secret" is clearly a test fixture value, note that and downgrade to Low.
